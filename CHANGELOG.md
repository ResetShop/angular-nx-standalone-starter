# Changelog

All notable changes to this starter are documented here. Forks should read this file before every upstream merge — it is the single source of truth for what changed upstream and whether any of those changes affect fork-owned code or require migration work.

The format is based on [Keep a Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/). Versioning cadence: **every fork-visible breaking change cuts a new version**; non-breaking changes accumulate under `## [Unreleased]` until the next breaking change forces a new release. A "fork-visible breaking change" is anything that requires a fork to take action during a merge — removed files, renamed APIs, changed schemas, restructured directories, or modified contracts. Pre-1.0 versions use the `1.0.0-beta.N` numbering convention while the fork-distribution model is being stabilized.

## [Unreleased]

### Added

- **Zod-validated env contract split into six domain-scoped sub-schema modules** under `apps/reference-app/src/api/config/`: `db.env.ts` (PG connection strings), `auth.env.ts` (PASETO, account lockout, bcrypt, cookies, CRON*SECRET), `email.env.ts` (EMAIL_PROVIDER + SMTP with the cross-field `superRefine` that requires SMTP_HOST/USER/PASS when EMAIL_PROVIDER=nodemailer), `http.env.ts` (PORT, CORS, BASE_HREF, IS_SERVERLESS — also exports `isServerless()` moved from `@resetshop/hono-core`), `app.env.ts` (NODE_ENV, APP_LANGUAGE, INTEGRATION_TEST_ADMIN_PASSWORD), and `cron.env.ts` (TOKEN_CLEANUP*\*). Each module is fully self-contained and exposes the same five symbols: an `XEnv` type, a `parseXEnv(rawEnv)` factory, a lazy-init `xEnv` Proxy populated from `process.env` on first property access, a `seedXEnv(overrides?)` test helper that writes directly into the cached singleton (bypassing `process.env`), and a `resetXEnv()` cache-clear helper for intra-file test isolation. Tolerant fallback via `.catch(default)` is preserved per-field (`AUTH_MAX_FAILED_ATTEMPTS`, `AUTH_LOCKOUT_DURATION`, `BCRYPT_COST`, `PORT`, `CORS_MAX_AGE`). Per-domain `*.env.spec.ts` files exercise every field via the corresponding `parseX` factory and the `seedX`/`resetX` helpers — no `process.env` mutation in tests. New `@config/*` path alias resolves to `apps/reference-app/src/api/config/*` in both `tsconfig.json` and `tsconfig.spec.json`. ([#354](https://github.com/ResetShop/angular-nx-standalone-starter/issues/354), [#356](https://github.com/ResetShop/angular-nx-standalone-starter/issues/356))
- **`docs/environment-variables.md`** — variables reference table plus four developer-choice delivery options for getting env vars into the process: (A) out-of-tree env file + Node `--env-file`, (B) IDE run configuration (WebStorm/VS Code with caveats about not committing inlined secrets), (C) shell session export, (D) `direnv` sourcing an out-of-tree file. Each option includes per-OS commands (macOS / Linux / Windows PowerShell — `$HOME` instead of `~` for PowerShell because PowerShell 5.x doesn't expand `~` for native exes). Optional centralized secrets managers (Doppler, Infisical, 1Password, dotenvx) are documented as drop-in wrappers (`<tool> run -- npm run dev`) but none ships in the starter. The new contract/delivery split is also added to `CLAUDE.md` as a new `## Environment Variables` section. ([#354](https://github.com/ResetShop/angular-nx-standalone-starter/issues/354))
- **Structural defenses against `.env` files reaching the working tree.** Three layered guards: (1) `scripts/check-no-env-files.mjs` is invoked from both `.husky/pre-commit` and the `npm run ci` script — it fails with a clear policy-violation message if any file named `.env` or `.env.*` exists at the repo root. Cross-platform via `fs.readdirSync('.')` — no shell-specific patterns. (2) New ESLint `no-process-env` block in `eslint.config.mjs` uses `no-restricted-syntax` to block any `MemberExpression[object.name="process"][property.name="env"]` across `apps/**/src/**/*.ts` and `packages/**/src/**/*.ts`, with a minimal four-entry allowlist: the sub-schema modules themselves (matched by `apps/**/src/api/config/*.env.ts`), the integration-test bootstrap files under `apps/**/src/api/integration/setup/**` (which run in a separate Node worker process and must mutate `process.env` before any sub-schema loads in that process), and the `test-utils.ts` files in `apps/**/src/` and `packages/util/` (re-listed only to preserve the loosened `vi.*` exemption — neither actually reads `process.env`). (3) `.claude/settings.local.json` adds Read/Write/Edit/Bash deny patterns on `./.env*` so Claude Code (and any agent honoring the settings) cannot read or modify a `.env` if one somehow appears. ([#354](https://github.com/ResetShop/angular-nx-standalone-starter/issues/354), [#356](https://github.com/ResetShop/angular-nx-standalone-starter/issues/356))
- **`PasetoConfig` DI-injected service config** at `apps/reference-app/src/api/services/paseto/paseto.config.ts` — extracts the five PASETO settings (`secretKey`, `issuer`, `accessTokenExpiry`, `refreshTokenExpiry`, `clockTolerance`) into a frozen interface produced by `createPasetoConfig()`. `PasetoService` constructor now takes `{ pasetoConfig }` via Awilix proxy injection instead of reading `process.env` directly. Tests construct services with a literal `PasetoConfig` object (no `process.env` mutation), which is what makes the "reject token generated with a different key" / "reject token with wrong issuer" cases work cleanly under the new frozen-singleton env model. Mirrors the existing `AuthConfig` pattern. ([#354](https://github.com/ResetShop/angular-nx-standalone-starter/issues/354))

The cleanup-tokens controller (`POST /api/auth/cleanup-tokens`) now reads its bearer-token secret via `container.cradle.authConfig.cronSecret` instead of `process.env['CRON_SECRET']` directly. `createAuthConfig` also gained an optional `AuthEnv` parameter (`createAuthConfig(source: AuthEnv = authEnv)`) so the env-to-AuthConfig field-mapping tests can drive specific values via `parseAuthEnv` without mutating `process.env`. ([#354](https://github.com/ResetShop/angular-nx-standalone-starter/issues/354), [#356](https://github.com/ResetShop/angular-nx-standalone-starter/issues/356))
- **`@resetshop/generators:ui-component` Nx generator** — scaffolds a shared UI library component under `packages/ui/src/lib/<name>/` with all project conventions pre-filled: `ChangeDetectionStrategy.OnPush`, `app-` element selector, `@reference "#tailwind-theme"` in styles, `clearAllMocks()` in `beforeEach`, and `parameters.docs.canvas.sourceState: 'shown'` in the Storybook meta. Inputs: `name` (required), `directory` (default `packages/ui/src/lib`), `exportFromIndex` (boolean, default `true`), `inlineTemplate` (boolean, default `true`), `inlineStyle` (boolean, default `true`). Invocation: `nx g @resetshop/generators:ui-component tooltip [--inlineTemplate=false] [--inlineStyle=false]`. The generators index in `packages/generators/generators.json` now exposes eight generators; `.claude/references/generators.md` and the README `Generators` table reflect the addition. Spec: `packages/generators/src/generators/ui-component/index.spec.ts` (15 tests covering substitution, sidecar emission, kebab-casing of compound names, idempotent index append, all four flag combinations, and a flag-invariance assertion on `.spec.ts`/`.stories.ts`). ([#348](https://github.com/ResetShop/angular-nx-standalone-starter/issues/348))
- **`custom-storybook/storybook-source-state` ESLint rule** (`packages/ui/eslint/storybook-source-state.js`) — fails CI when any `*.stories.ts` file omits `parameters.docs.canvas.sourceState: 'shown'`. Walks the `ExportDefaultDeclaration` and resolves both inline-object exports (`export default { ... }`) and variable-referenced exports (`const meta = { ... }; export default meta`); cross-module imports of meta objects are silently skipped (safer than false-positives). Reports five distinct message IDs (`missingParameters`, `missingDocs`, `missingCanvas`, `missingSourceState`, `wrongSourceState`) for precise diagnostics. Registered in `eslint.config.mjs` under a new `storybook-source-state` named config block targeting `**/*.stories.ts`. Covered by a `RuleTester` spec at `packages/ui/eslint/storybook-source-state.spec.js` (8 cases — 3 valid, 5 invalid). **Forks with `*.stories.ts` files that omit `sourceState: 'shown'` will start failing `npm run lint` on the next upstream merge.** Migration: add `parameters: { docs: { canvas: { sourceState: 'shown' } } }` to each affected meta (same shape the `ui-component` generator scaffolds). ([#348](https://github.com/ResetShop/angular-nx-standalone-starter/issues/348))
- **`/issue-workflow` Claude Code skill** (`.claude/skills/issue-workflow/SKILL.md`) — orchestrates the full 6-phase issue resolution lifecycle (Setup → Plan → Implement → Review → Fix → Ship) as a single repeatable command invoked with `/issue-workflow <issue-url>`. Derives branch name from the issue title via `gh issue view`, delegates to `plan-writer` and `code-reviewer` agents at the appropriate phases, enforces atomic commits with `[#<issue>] - <description>` messages, and pauses for user approval after Plan and Review. Hard constraints embedded: no direct `nx` commands, no `cd`-prefixed git, no PR before CI + review, no skipped Plan phase. Documented in CLAUDE.md under "Issue Workflow Skill". ([#349](https://github.com/ResetShop/angular-nx-standalone-starter/issues/349))
- **Interactive skill installer** (`scripts/setup-skills.mjs`) — postinstall hook that auto-installs third-party Claude Code skills on `npm install`. In interactive terminals, presents an Nx-style multi-select prompt (arrow-key navigation, spacebar toggle, enter to confirm); in CI/piped environments, auto-installs all default-selected skills silently. Supports two install strategies: `clone` (shallow git clone into `.claude/skills/<name>/`) and `command` (runs a shell command, e.g. `npx nx configure-ai-agents`). Currently ships with the Angular skill (`angular/skills`); Nx skill deferred to [#351](https://github.com/ResetShop/angular-nx-standalone-starter/issues/351) pending README reconciliation. - **Single-app starter layout.** The original root-level `src/`, `project.json`, `tsconfig.app.json`, `tsconfig.spec.json`, `playwright.config.ts`, `vitest.config.ts`, `vitest.integration.config.ts`, and `tools/` directories have been removed. Their content lives under `apps/reference-app/` or `packages/*/eslint/` now. **Fork action:** any fork-local imports via the old root-relative paths must be updated to use `apps/reference-app/...` paths or `@resetshop/*` aliases.
- **`apps/reference-app/src/api/openapi-app.ts`** and **`apps/reference-app/src/api/utils/environment.ts`** — replaced by `@resetshop/hono-core` exports.
