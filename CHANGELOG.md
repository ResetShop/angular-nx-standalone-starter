# Changelog

All notable changes to this starter are documented here. Forks should read this file before every upstream merge — it is the single source of truth for what changed upstream and whether any of those changes affect fork-owned code or require migration work.

The format is based on [Keep a Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/). Versioning cadence: **every fork-visible breaking change cuts a new version**; non-breaking changes accumulate under `## [Unreleased]` until the next breaking change forces a new release. A "fork-visible breaking change" is anything that requires a fork to take action during a merge — removed files, renamed APIs, changed schemas, restructured directories, or modified contracts. Prior to 1.0.0, versions followed a `1.0.0-beta.N` naming convention while the fork-distribution model was being stabilized; all releases from 1.0.0 onward follow semantic versioning.

Two repo labels are introduced — **`✍️ draft`** (English of cuentoneta's `✍️ bosquejo`, marking freshly-proposed issues pending refinement) and **`🧩 initiative`** (a planning tier above `📦 epic`) — and the unused literature-specific **`📜 content`** label is removed. The existing `.github/ISSUE_TEMPLATE/config.yml` (security advisory + contribution-policy links, `blank_issues_enabled: true`) is preserved unchanged, and `.github/pull_request_template.md` gains a `Closes #<issue-number>` linking reminder in its Summary section. Documentation/config only — no runtime, API, or schema change. **Fork migration:** forks inherit the templates and PR template automatically; fork maintainers should create the `✍️ draft` and `🧩 initiative` labels in their own repo (or retune the templates' `labels:` arrays to match their own label names) since GitHub silently drops default labels that don't exist in the target repo. - **CI falls back to Nx's local cache when Nx Cloud credits are exhausted ([#529](https://github.com/ResetShop/angular-nx-standalone-starter/issues/529)).** Every CI job runs Nx tasks with `NX_CLOUD_ACCESS_TOKEN` set (inherited from the workflow-level `env`), so when the Nx Cloud free credits run out the workspace is disabled and Nx aborts **every** task with a non-zero exit code — even though the underlying task (tests, lint, build) would pass — blocking every PR until credits reset. A new wrapper `.github/scripts/nx-run.sh` runs `npx nx <task>` and, **only** when the failure output matches a "workspace is disabled / Nx Cloud limit/quota/exceed" pattern, retries once with `NX_NO_CLOUD=true` so the task runs against Nx's local cache instead of failing the pipeline; any other failure (a broken test, red lint, etc.) is propagated unchanged. All eight Nx steps in `.github/workflows/ci.yml` are routed through the wrapper (`test`, `typecheck`, `lint`, `stylelint`, `test-integration`, `e2e`, `build`, `storybook`) — `test-integration`'s underlying Nx command is inlined through the wrapper rather than going via `npm run test:integration`, matching how `e2e`/`build`/`stylelint` already inline their Nx commands. A new root `.gitattributes` pins `*.sh` to `text eol=lf` so the script is not CRLF-corrupted when edited on a Windows checkout (CI runs it with bash on Linux). Ported from [cuentoneta/cuentoneta#1628](https://github.com/cuentoneta/cuentoneta/pull/1628); unlike that repo (which rides a pre-existing per-job `.nx/cache` `actions/cache`), this repo caches the whole repo keyed by `github.sha`, so the fallback runs cold locally — it still unblocks CI, just without an accelerated local cache. CI-infrastructure only; no runtime, API, or schema change. **Fork migration:** none required; forks inherit the resilience automatically. ([#529](https://github.com/ResetShop/angular-nx-standalone-starter/issues/529))
- **`railway.json` deploy config (build/start/healthcheck as code) ([#514](https://github.com/ResetShop/angular-nx-standalone-starter/issues/514)).** Added a root `railway.json` that codifies the Railway deployment instead of leaving it only in the service UI: `build.buildCommand` (`npm run build` → `dist/reference-app/server/server.mjs`), `deploy.startCommand` (`npm run serve:ssr`), a `deploy.healthcheckPath` of `/api/health/v1` (the app's public health endpoint), and an `ON_FAILURE` restart policy with a bounded retry count. The healthcheck gates a deploy on a reachable database, so a misconfigured deploy (e.g. a wrong `PG_CONNECTION_STRING`) can no longer replace a healthy one, and the bounded retries cap the restart loop the startup `verifyHealth` would otherwise produce. Makes deploys reproducible across prod and PR-preview environments and inherited by forks. **Note:** Railway config-as-code takes precedence over the service UI — the committed `buildCommand`/`startCommand` match the current settings, and `healthcheckPath` is additive. ([#514](https://github.com/ResetShop/angular-nx-standalone-starter/issues/514))

### Changed

- **Comment-quality policy: no issue-number or before/after references in code comments, plus a codebase sweep and an enforcement guard ([#499](https://github.com/ResetShop/angular-nx-standalone-starter/issues/499)).** Code comments must describe present-state rationale only — what the code does and why it must be this way now — never the history of how it got there. The policy is recorded in `CLAUDE.md` (a new "Code Comment Quality" subsection plus a Hard Constraints table row) and in `.claude/references/coding-agent-policies.md` (new Section 6), banning GitHub issue/PR number tokens (`#<n>`), before/after-issue framing (`Since #497…`, `Pre-#331 this was…`), and "formerly X" / "moved here from" framing from `.ts` / `.spec.ts` / `.stories.ts` comments. The policy also states positively that **PRs and issues are the canonical home for change-narrative documentation** — the why-it-changed, the before/after story, and the issue/PR cross-references live in the PR description and the issue thread (a code comment documents the present invariant; the PR and issue document the change that produced it). Swept 15 comment sites across 13 source files, rewriting each as a present-state description (e.g. `// regression guard for #471` → a plain-English statement of the singleton-bridge invariant). Added `scripts/check-no-issue-refs-in-comments.mjs` — modelled on `check-no-env-files.mjs`, scanning every `.ts` file under `apps/` and `packages/` for a `#[1-9]\d{2,}` token in a `//` or block-comment line — wired into the `.husky/pre-commit` hook and both the `ci` and `ci:verify` npm scripts. `CHANGELOG.md`, `docs/`, `README.md`, and `.claude/` are exempt by path (hyperlinked issue references there are durable cross-references). No runtime behavior change. **Fork migration:** none required, but forks carrying issue-number references in `.ts` comments will fail the new pre-commit/CI guard after merging — rewrite those comments as present-state rationale (or, for genuinely durable cross-references, move them to `CHANGELOG.md` or a `docs/` file). ([#499](https://github.com/ResetShop/angular-nx-standalone-starter/issues/499))
- **`@reference "#tailwind-theme"` is now auto-injected at build time and removed from component source ([#523](https://github.com/ResetShop/angular-nx-standalone-starter/issues/523)).** Tailwind v4 requires each isolated component stylesheet that uses `@apply` to declare `@reference` so the directive resolves against the theme; previously every such stylesheet carried the line in source, and the `ui-component` generator emitted it into every new component. A new root PostCSS plugin `postcss-inject-tailwind-reference.js` (registered first in `.postcssrc.json`, before `@tailwindcss/postcss`) prepends `@reference "#tailwind-theme"` at build time to every stylesheet that uses `@apply` — covering all three pipelines that read `.postcssrc.json` (the Angular esbuild builder, Storybook's webpack `postcss-loader`, and Vite/Vitest). Injection is gated on `@apply` because that is the only thing the reference resolves: global/theme-emitting stylesheets (e.g. `styles.css`, which `@import`s the theme to **emit** it) are deliberately skipped, since referencing the theme there double-references it and suppresses its emitted `@theme` tokens (e.g. the `--font-*` variables). The plugin is idempotent (skips a stylesheet that already declares `@reference`) and excludes theme-entry files (`tailwind.config.css`, `accessibility.css`), which must keep `@reference "tailwindcss"` to avoid a circular self-reference. The plugin and `.postcssrc.json` are registered as nx `sharedGlobals` so build/test/storybook caches invalidate when they change. The directive was removed from all 8 component source stylesheets (`dashboard`, `nav-item`, `sidebar`, `button`, `form-field` inline `styles:`, and `toast-notification.css`/`select.css`/`combobox.css` sidecars), and the `ui-component` generator (`EXTERNAL_STYLE_BODY`, the inline template, the spec, and `generators.md`) no longer emits it. Payload is unaffected — `@reference` produces no output CSS. **Migration:** forks that author component stylesheets no longer write `@reference "#tailwind-theme"` (it is injected automatically); a new file `@import`ed into `tailwind.config.css` that uses `@apply` must be added to the plugin's theme-entry exclusion list (a miss surfaces as a loud Tailwind circular-reference build error). ([#523](https://github.com/ResetShop/angular-nx-standalone-starter/issues/523))
- **`no-process-env` ESLint allowlist tightened to least-privilege; integration tests routed through the env schemas ([#501](https://github.com/ResetShop/angular-nx-standalone-starter/issues/501)).** The `no-process-env-allowlist` block in `eslint.config.mjs` previously lifted the `process.env` restriction for three directory wildcards (`apps/**/src/api/config/**/*.ts`, `apps/**/src/api/integration/**/*.ts`, `apps/**/src/db/**/*.ts`), silently permitting raw `process.env` access in every file under those trees. It is now an explicit per-file list (no `/**/` directory wildcards — `**` remains only as a multi-app prefix): `env-utils.ts`, `env-utils.spec.ts`, `config/runtime.ts`, `config/runtime.spec.ts`, `test-setup.ts`, `integration/setup/global-setup.ts`, `integration/setup/env-helpers.ts`, and `db/seed.ts`. To get there: a new `apps/reference-app/src/api/config/runtime.ts` exports `isInteractive()` (reads `CI` + `stdin.isTTY`; `seed.ts` now uses it); `integration/setup/env-helpers.ts`'s `configureEnvVars` now seeds the typed caches via `seedDbEnv`/`seedTokenEnv`/`seedEmailEnv`/`seedPasswordEnv`/`seedCronEnv` (the same Zod parse the proxies run) instead of writing `process.env`; `db-helpers.ts`, `auth-helpers.ts`, and `login.integration.spec.ts` now read `appEnv.INTEGRATION_TEST_ADMIN_PASSWORD` / `dbEnv.PG_TEST_CONNECTION_STRING`; and the generic proxy `process.exit(1)` FATAL test moved from `db.env.spec.ts` into a new `env-utils.spec.ts`. `global-setup.ts` keeps its raw `process.env['PG_TEST_CONNECTION_STRING']` write (worker child processes inherit it via the OS environment — in-memory seed caches do not cross process boundaries), and `env-helpers.ts` keeps `loadEnvFile` as the single sanctioned integration `.env` delivery boundary (its `getTestConnectionString` reads `process.env` raw because it runs before the caches are seeded). **Migration:** forks that added direct `process.env` reads anywhere under `src/api/config`, `src/api/integration`, or `src/db` other than the eight listed files will now fail `npm run lint` on the next merge — route the value through the appropriate `<domain>Env` proxy (or `seed<Domain>Env` in tests), or add a justified explicit entry to the allowlist. ([#501](https://github.com/ResetShop/angular-nx-standalone-starter/issues/501))
- **Dashboard header chrome tokenized — hardcoded grays replaced with semantic theme tokens ([#520](https://github.com/ResetShop/angular-nx-standalone-starter/issues/520)).** The dashboard header chrome used hardcoded Tailwind palette colors and `dark:`-only one-offs instead of the design system's semantic theme tokens. `apps/reference-app/src/app/components/header/header.ts`'s host binding carried a dangling `dark:text-gray-50` (a dark-only text color with no light-mode counterpart), and `apps/reference-app/src/app/pages/dashboard/dashboard.ts`'s `<header appHeader>` element used `border-gray-200` / `dark:border-white/10` / `dark:bg-black/95`. These now use `text-foreground`, `border-border`, and `bg-background` — the oklch-backed tokens defined in `tailwind.config.css` that adapt to light/dark automatically, matching the rest of the app (`breadcrumb`, `landing-header`). The swap is visually exact in both modes: `--foreground` dark ≈ `gray-50`, `--border` dark = `white/10`, and `--background` dark ≈ `black/95` (chosen over `bg-card`, whose dark value `20.5%` is lighter than the prior `black/95` ≈ `14.5%`). The header-chrome wrapper elements in `header.stories.ts` were repointed to the same tokens so Storybook renders the header as the app now does; the surrounding sidebar/main/label simulation chrome in those stories is left untouched (out of scope, per the issue). Surfaced by the [#489](https://github.com/ResetShop/angular-nx-standalone-starter/issues/489) host-migration code review, which carried the `dark:text-gray-50` over verbatim. - **Host-level `:host { @apply … }` styles migrated to the `host: { class }` decorator binding, and the `@reference` target standardized ([#489](https://github.com/ResetShop/angular-nx-standalone-starter/issues/489)).** Components whose host styling was unconditional, host-only, and purely utility-based now declare those utilities via `host: { class: '…' }` instead of an encapsulated `:host { @apply … }` block, deleting the now-empty `styles` blocks and their orphaned `@reference` lines: `brand`, `header`, `immersive-panel`, the four auth pages (`login`, `change-password`, `reset-password`, `reset-password-confirm`), and the host line of `loading-spinner` (its `.spinner-ring`/`@keyframes` rules remain). `nav-item` is partially migrated — its unconditional `cursor-pointer text-sm` moves to `host: { class }` while the conditional `:host(.collapsed)` and descendant rules stay. **Behavioral note:** `:host { @apply }` emits view-encapsulated CSS (`[_nghost-…]` specificity); `host: { class }` emits global utility classes at base specificity — equivalent for these layout utilities since nothing overrides them on the host. The `@reference` target string is standardized to `"#tailwind-theme"` for all isolated consumer stylesheets (`dashboard`, `button`, `toast-notification.css`, plus `nav-item`), with one structurally-required exception: `packages/ui/src/theme/accessibility.css` is `@import`ed into the theme entry (`tailwind.config.css`) and so must keep `@reference "tailwindcss"` (referencing `#tailwind-theme` from inside the theme it composes is circular). A `@reference "#tailwind-theme"` must also sit at the top of its stylesheet, never nested in a rule, because the theme entry carries `@source` (which cannot be nested) — `dashboard`'s `@reference` was hoisted out of its `:host` block accordingly. The non-host cases (`sidebar`, `button`/`form-field` projected children, the standalone `select.css`/`combobox.css`/`toast-notification.css`) intentionally keep `@apply`/`@reference` — `host: { class }` cannot express projected-child, conditional, or non-utility styling. Full project-wide `@reference` elimination (build-time auto-injection or a raw-CSS rewrite) was evaluated and deferred to [#523](https://github.com/ResetShop/angular-nx-standalone-starter/issues/523), which carries the findings; the migration's measured payload delta was +239 B / +0.5% (payload was never the motivation). No fork migration required — the rendered output is unchanged. ([#489](https://github.com/ResetShop/angular-nx-standalone-starter/issues/489))
- **Standalone DB scripts consolidated onto the canonical Drizzle connector ([#502](https://github.com/ResetShop/angular-nx-standalone-starter/issues/502)).** `apps/reference-app/src/db/seed.ts` and `sync-permissions.ts` previously each built their own `new Pool()` + `drizzle(pool)` instance (and `seed.ts` derived an ad-hoc `type Tx` from it). This removes the duplicated connection wiring and the local transaction type, and the scripts gain the connector's idle-client `error` listener (a connection drop/restart mid-run no longer risks an unhandled-error process crash). The connector is created inside each script's entry function (function scope, not module scope), and both scripts now close their connection pool via `db.$client.end()` before exiting (the `process.exit` calls moved to the entry point so the `finally` runs). Runtime behavior is otherwise unchanged — both scripts use the same core `select`/`insert`/`transaction` calls, for which the schema-typed connector is a superset. The connector's `@schema/*` / `@config/*` / `@resetshop/util` path-alias imports resolve under the scripts' `tsx --tsconfig apps/reference-app/tsconfig.json` invocation (verified). `e2e/setup/db-seed.ts` intentionally stays on direct `bcrypt`/`pg` wiring (Playwright's loader resolves these aliases unreliably). No migration needed for forks. ([#502](https://github.com/ResetShop/angular-nx-standalone-starter/issues/502))

- **Admin seed credentials are now operator-supplied, not hardcoded ([#491](https://github.com/ResetShop/angular-nx-standalone-starter/issues/491)).** `apps/reference-app/src/db/seed.ts` previously created an admin with a known email (`admin@sistema.com`) and a committed bcrypt hash — offline-crackable once public, and inherited by every deployer who ran `npm run drizzle:seed` unchanged. Credentials are now resolved in precedence order: (1) `SEED_ADMIN_EMAIL` + `SEED_ADMIN_PASSWORD` env vars (CI/Docker/prod-safe, non-interactive); (2) an interactive TTY prompt with masked password input and re-prompt on invalid entry; (3) fail-fast with a clear, actionable error. `SEED_ADMIN_FIRST_NAME` / `SEED_ADMIN_LAST_NAME` are optional (default `Admin` / `User`) — all four are declared in the `app` env sub-schema. The password is hashed at runtime via the app's `createPasswordHasher()` (cost from `passwordEnv.BCRYPT_COST`). The resolution logic lives in a new, fully unit-tested `seed-admin-credentials.ts`. The seed stays idempotent — an existing admin row is left untouched. **Migration:** set `SEED_ADMIN_EMAIL` and `SEED_ADMIN_PASSWORD` (12–128 chars) before running `npm run drizzle:seed` in CI/non-interactive contexts, or run it in a terminal to be prompted; existing fork databases are unaffected (the seed never overwrites existing rows). ([#491](https://github.com/ResetShop/angular-nx-standalone-starter/issues/491))
- **`postinstall` no longer runs database migrations or the seed ([#491](https://github.com/ResetShop/angular-nx-standalone-starter/issues/491)).** `npm install` / `npm ci` previously ran `npm run drizzle:push-migrations && npm run drizzle:seed` automatically, which failed the install on any machine without `PG_CONNECTION_STRING` set and would hang on the seed's interactive prompt in piped contexts. The hook now runs `node scripts/setup-skills.mjs` only; the database steps are explicit. **Migration:** any CI/CD pipeline, Dockerfile, or fork that relied on `npm install` to apply migrations must add an explicit `npm run drizzle:push-migrations && npm run drizzle:seed` step (after configuring `PG_CONNECTION_STRING` and the `SEED_ADMIN_*` vars) following the install step. ([#491](https://github.com/ResetShop/angular-nx-standalone-starter/issues/491))

Previously the strategy passed the raw `route.title` value (an i18n key like `USERS.PAGE.TITLE`) straight to `Title.setTitle()`, so the browser tab read `USERS.PAGE.TITLE | <applicationName>` instead of `Users | <applicationName>`. Also added an `effect` that watches `Translation.currentLanguage` and re-applies the title when the active language changes between navigations (without it the tab title stayed frozen in whichever language was active at the last route change).
- **`apps/reference-app/src/contracts/auth/auth.schemas.ts`** — `meResponseSchema.email` was `z.string()` while `authUserSchema.email` was `z.email()`. Collapsing `meResponseSchema` into `authUserSchema` (see Changed entry above) tightens email validation on `/api/auth/me` to match `/api/auth/login`. ([#322](https://github.com/ResetShop/angular-nx-standalone-starter/issues/322))
- **Sidebar nav flicker during logout** — `provideAuth()` (`apps/reference-app/src/app/providers/auth/auth.provider.ts`) now accepts opt-in features in the `provideRouter`/`provideHttpClient` style. The new `withNavigationPermissionCheck()` feature wires the navigation library's `NAVIGATION_PERMISSION_CHECK` token through `AuthStore.currentUser`, and **returns `true` for any permission when no user is logged in**. The previous inline factory in `app.config.ts` re-evaluated `Navigation.sections` to a permission-stripped list the instant `authStore.logout()` cleared `currentUser`, producing a one-frame visible flicker before the route deactivated and the sidebar unmounted. Skipping the filter when no user is logged in eliminates the recomputation; route-level access is still enforced by `permissionGuard`, so this carries no security impact. **Migration for forks:** if your `app.config.ts` calls `provideAuth()` and you want the navigation permission check, change it to `provideAuth(withNavigationPermissionCheck())` and remove any inline `NAVIGATION_PERMISSION_CHECK` provider you may have copied. `provideAuth()` with no features is still valid and registers only the `AuthApi` token.

## 1.0.0-beta.2 — 2026-04-07

Adds the upstream CI guard layer that protects the starter contract on incoming PRs.

### Added

- **`.github/workflows/upstream-guards.yml`** — single workflow with two jobs:
  - **`boundary-guard`** — fails if a PR touches any path under `apps/` other than `apps/reference-app` or its subdirectories. Sibling-named directories (e.g. `apps/reference-app-staging/`) are deliberately treated as offending. Bypass label: `allow-app-change`.

## 1.0.0-beta.1 — 2026-04-07

Adds the starter/app boundary enforcement layer on top of the structural restructure shipped in `1.0.0-beta.0`.

### Added

- **Nx tag scheme `scope:starter` / `scope:app`** — all upstream-owned projects (`packages/*` and `apps/reference-app`) carry `scope:starter`; fork-generated apps emit `scope:app` from the schematic. The schematic's tag rewrite now actually fires against the renamed workspace.
- **ESLint `@nx/enforce-module-boundaries` `depConstraints` for `scope:*`** — `scope:starter` projects may only depend on other `scope:starter` projects; `scope:app` projects may depend on both `scope:starter` and `scope:app`. Stacks with the existing `type:*` constraints.
- **ESLint cross-boundary relative-import bans** — two new flat-config blocks (`no-cross-boundary-relative-imports-from-packages` and `no-cross-boundary-relative-imports-from-apps`) using depth-agnostic `^(\.\./)+(apps|packages)/` regex patterns. Forces consumers to use the `@<scope>/*` package aliases instead of relative paths across the boundary.

### Changed

- **`nx.json` `defaultBase` changed from `master` to `main`** — matches the actual upstream branch name; fixes silent breakage of `nx affected` and Nx Cloud CIPE diffs. **Fork action:** if your fork's primary branch is not `main`, change `defaultBase` once to match your branch name after merging this version.

### Removed

- **`nx.json` `defaultProject`** — relied on by direct `nx` CLI calls which are forbidden by CLAUDE.md's command policy. Removing it eliminates a fork-merge conflict surface and forces explicit project naming in every command.

## 1.0.0-beta.0 — 2026-04-06

This is the first tagged version of the starter under the fork-distribution model. Forks created after this version should track the changelog entries above as the canonical record of what has changed since their initial fork point.

### Added

- **Fork-distribution model.** The repository is now designed to be forked rather than consumed as a published package set. Starter code lives in `packages/*` and `apps/reference-app`; fork apps live in `apps/<your-app>`.
- **`@resetshop/generators:app` schematic** — clones `apps/reference-app` into `apps/<slug>` for new app creation. Invoked via `npm run generate:app -- --name="My App"`. Includes 20 unit tests and a smoke-test pass against the live workspace.
- **`docs/forking.md`** — full forking workflow documentation: ownership boundaries, initial setup, app creation, upstream merge process with conflict resolution conventions, the changelog contract, and what NOT to do.
- **README forking section** — quickstart snippet pointing at `docs/forking.md`.
- **`packages/hono-core` exports `isServerless()`** — moved from a per-app helper file in `apps/reference-app/src/api/utils/environment.ts` so backend modules across multiple apps can share the same runtime check.
- **Uniform CI target set** — every project under `packages/*` and `apps/*` now exposes the targets it legitimately needs (`build`, `test`, `typecheck`, `lint`, `stylelint`, `build-storybook` where applicable).
- **`CLAUDE.md` Canonical App Creation Workflow section** — explicit policy that new apps must always be created via the schematic; direct hand-copying of `apps/reference-app` is forbidden.
- **Coding agent collaboration policy** — `.claude/references/coding-agent-policies.md` codifies the rule against shortcut framings on solo-maintainer assumptions, plus other anti-patterns. Referenced from `CLAUDE.md` as a hard constraint.

### Changed

- **Monorepo restructured** under `packages/*` and `apps/reference-app/src/`. UI components live in `@resetshop/ui`, Angular providers in `@resetshop/angular-core`, Hono backend infrastructure in `@resetshop/hono-core`, framework-free utilities in `@resetshop/util`, and Nx generators in `@resetshop/generators`. All app-specific code (stores, providers, API modules, drizzle schemas, contracts) lives under `apps/reference-app/src/`.
- **Tailwind config moved to workspace root.** `tailwind.config.css` now lives at the workspace root and globs into both `apps/*` and `packages/*` via `@source` directives, so all projects pick up the same theme.
- **Backend imports migrated from relative `'../../openapi-app'` to `@resetshop/hono-core`** across the reference app's API controllers and repositories. Forks consuming the reference template see no behavioural change but should use the package alias for any new backend code.
- **`package.json` `dependencies` and `devDependencies` strictly alphabetized** — minimizes conflict surface for forks adding their own dependencies.

### Removed

- **Single-app starter layout.** The original root-level `src/`, `project.json`, `tsconfig.app.json`, `tsconfig.spec.json`, `playwright.config.ts`, `vitest.config.ts`, `vitest.integration.config.ts`, and `tools/` directories have been removed. Their content lives under `apps/reference-app/` or `packages/*/eslint/` now. **Fork action:** any fork-local imports via the old root-relative paths must be updated to use `apps/reference-app/...` paths or `@resetshop/*` aliases.
- **`apps/reference-app/src/api/openapi-app.ts`** and **`apps/reference-app/src/api/utils/environment.ts`** — replaced by `@resetshop/hono-core` exports.
